## 一、Agent 视角的当前问题（本 session 自检 + 历史 dogfooding）

我作为长期被 karma 监督的 Agent，记录看到当前规则文本时的真实第一反应：

### 问题 1：警示词激活防御反应而非反思

**当前文本样例**（`karma/sticky.py:format_for_injection`）：

```
[karma sticky — 用户最高优先级方向，请始终遵守]
3. 用直白中文。不用英文技术术语（F1 / precision 等）。 ⚠️ 上次违反！
```

**Agent 第一反应**：「⚠️ 红警示 + '违反' 字眼 → 我被指责了 → 防御性补偿（堆中文字凑比例 / 找借口合理化用 jargon）」

**心理学根据**：威胁式语气激活 fight-or-flight，Agent 倾向「绕过 / 表面满足」而非「反思根因」。

### 问题 2：精确数字阈值让 Agent 优化 metric 而非真改

**当前文本样例**（`chinese_plain.py` trigger）：

```
触发: 自然语言中文占比 34% < 40%
```

**Agent 第一反应**：「34% → 让它 ≥ 40% 就过线」加几个汉字虚词凑数。

**根因**：精确数字给了「优化目标」，但目标是 metric 而非 metric 背后的「用户能听懂」。

### 问题 3：反思式提问被「真合理」字眼漏掉

**当前文本样例**（`chinese_plain.py:145` suggested_fix）：

```
想想这段英文比例是有真合理原因（引用项目专名 / 标准技术词 / 复制粘贴他人内容）
还是习惯堆 jargon？合理就保留；不合理就换汉字。
```

**Agent 第一反应**：「我用『API / commit / merge』都是标准技术词 → 真合理 → 保留」。

**根因**：「合理化通道」开太大，Agent 容易抓「合理就保留」当借口。

### 问题 4：suggested_fix 全是「不要 / 禁止」类负面命令

**当前文本样例**（`non_blocking.py:133` / `evidence.py:123`）：

```
不要 sleep 阻塞前端。用 run_in_background=True 启动任务...
不要用『应该』掩盖不确定。明说不知道...
```

**Agent 第一反应**：「立刻避免那个动作」但没有正面行为模板可学。

**根因**：纯禁止反馈不教 Agent「下次怎么做对」，只教「下次别做错」— 错的方式可能无穷多种。

### 问题 5：「本 turn 立即按 fix 改进，不要再犯同样错误」类惩罚式收尾

**当前文本样例**（`user_prompt_submit.py:117`）：

```
本 turn 立即按 fix 改进，不要再犯同样错误。
```

**Agent 第一反应**：教师批评学生的语气，激发「应付完事」而非「真心改」。

### 问题 6：注入文本通篇缺少「用户是真实的人」共情

**当前文本**通篇用「sticky / 违反 / 触发 / 拦截」类工具化术语，Agent 看不到「用户是非技术身份听不懂 jargon」「用户期待跟你合作不是审你」这类真人视角。

**根因**：karma 设计成「规则系统」而非「合作默契」— Agent 把它当 lint 工具而非用户共情提醒。

## 二、设计原则（双视角同时满足）

| 用户视角 | Agent 视角 | 设计原则 |
|---|---|---|
| 规则执行准确 | 第一反应不抗拒 | **共情而非命令** — 让 Agent 看到用户而非规则 |
| 不漏拦真违反 | 不容易找借口合理化 | **明确但不给逃生口** — 反思式但有具体例子 |
| 不过度刺激 | 不激活防御反应 | **去警示词** — 删「⚠️」「违反」「不要再犯」类红词 |
| 让 Agent 真改行为 | 知道下次怎么做对 | **加正面行为模板** — 不只禁止还示范 |
| 持续合作长期演化 | 把 karma 当伙伴不是审判 | **「合作默契」而非「规则系统」** |

## 三、具体改造方案（4 个维度）

### 维度 A：sticky.yaml `preference` 文本风格

**优化点**：把规则陈述变成「用户委托语气」— Agent 感受到「这是一位真人朋友对我的期待」。

**对照样例**（chinese-plain-no-jargon）：

| 当前 | 优化后 |
|---|---|
| `用直白中文。不用英文技术术语（F1 / precision / supervisor / oracle 等）。用「综合得分 / 精度 / 召回率」等完整汉字。必须用英文专有名词时短解释一下。用类比解释复杂概念，不堆 jargon。` | `跟你协作的用户是非技术身份，他要的是听得懂的汇报。把「precision / dispatcher / mutex」换成「精度 / 分发器 / 互斥锁」让他读起来不卡。真技术专名（项目名 / 论文术语 / 业界缩写）保留，但首次出现配中文短解释。用类比解释复杂概念，比堆 jargon 更显你懂。` |

**优化点**：
1. 起手提「用户是真人 + 他有具体需要」
2. 用「让他读起来不卡」「显你懂」类共情语
3. 不报数字（去掉「< 40%」类）
4. 保留例外通道但锚到具体场景（「真技术专名」+ 限定项目名 / 论文术语 / 业界缩写）

**对照样例**（long-term-fundamental）：

| 当前 | 优化后 |
|---|---|
| `用最根本、最长期、最普适、最优雅的方案。不打补丁、不硬编码、不为追短期 KPI 牺牲长期质量。深挖到真根因再动手；遇到难题先想「最干净的解法是什么」而非「最快糊过去」。` | `用户相信你能深挖根因。遇到难题他希望你先停下想「最干净的解法是什么」而不是「最快糊过去」。短期补丁 / 硬编码 / 跳验证 flag 都是「以后会还的债」— 他愿意为长期质量等你多想几分钟。如果方案分歧大就提出来跟他对齐，不要自己挑短期路再说。` |

**优化点**：
1. 用「用户相信你能 X」替代「用最 X 方案」（强调信任关系）
2. 「他愿意为长期质量等你多想几分钟」给 Agent 心理安全感（不催赶）
3. 加「提出来跟他对齐」给沟通通道（不闷头硬干）

---

### 维度 B：check `suggested_fix` 文本风格

**优化点**：从「禁止 + 命令」改「示范 + 共情解释」。

**对照样例**（chinese_plain.py:145 中文比例触发）：

| 当前 | 优化后 |
|---|---|
| `想想这段英文比例是有真合理原因（引用项目专名 / 标准技术词 / 复制粘贴他人内容）还是习惯堆 jargon？合理就保留；不合理就换汉字（如「精度」「召回率」「分发器」等）。` | `本段读起来用户可能要停下查几次「这词什么意思」。看看哪些英文是「项目名 / 论文术语」（这种保留 + 首现加中文解释），哪些是「随手用了英文」（换成精度 / 召回率 / 分发器 等汉字）。目标不是中文凑比例，是让用户读完不用查词。` |

**优化点**：
1. 把 metric 改成「用户阅读体验」（不是凑 ≥ 40%）
2. 明确分两类用法（项目名保留 vs 随手英文换）
3. 收尾用「目标是 X 不是 Y」澄清意图，避免 Agent 抓 metric 当借口

**对照样例**（non_blocking.py:133 sleep 阻塞）：

| 当前 | 优化后 |
|---|---|
| `不要 sleep 阻塞前端。用 run_in_background=True 启动任务，并行做其他事。` | `sleep 期间用户等你输出，体验是「卡了」。改成 run_in_background=True 启动任务，然后立刻推进下一件能做的（读相关文件 / 起另一个子 Agent / 设计下一步）— 任务完成时你会被通知到。` |

**优化点**：
1. 「用户等你输出，体验是『卡了』」用户视角共情
2. 给「立刻推进下一件能做的」具体行为模板 + 3 个例子
3. 「任务完成时你会被通知到」消除 Agent 焦虑（怕忘掉任务结果）

**对照样例**（evidence.py:123 弱化词）：

| 当前 | 优化后 |
|---|---|
| `不要用『应该』掩盖不确定。明说不知道，或者跑测试确认。` | `「应该」「可能」让用户分不清你是查过还是猜的。如果你是真不确定，明说「我不确定 X 因为 Y，可以跑 Z 验证下吗」— 这比错装确定更建立信任。如果是确定的就直说，去掉「应该」让信息清晰。` |

**优化点**：
1. 「让用户分不清你是查过还是猜的」用户视角痛点
2. 「明说『我不确定 X 因为 Y，可以跑 Z 验证下吗』」给具体话术模板
3. 「比错装确定更建立信任」给行为长期回报理由

### 维度 C：user_prompt_submit 头部注入格式

**当前格式**（`sticky.py:format_for_injection`）：

```
[karma sticky — 用户最高优先级方向，请始终遵守]
1. 用最根本、最长期、最普适、最优雅的方案。
   不打补丁、不硬编码、...
3. 用直白中文。不用英文技术术语（F1 / precision 等）。 ⚠️ 上次违反！
...
```

**优化后**：

```
[karma — 你跟用户的长期默契]
跟你协作的是一位非技术身份用户，他列出了几条长期最看重的方向。
这不是规则也不是审判 — 是他希望跟你建立的协作默契。

1. 用户相信你能深挖根因。遇到难题先想「最干净的解法」而不是「最快糊过去」...
3. 跟你协作的用户是非技术身份，他要的是听得懂的汇报...
   〔上一回应这条偏离了，本 turn 看看能否更对齐〕
```

**关键改动**：
1. 头部从「请始终遵守」改「合作默契」
2. 加一行「这不是规则也不是审判」破除监督感
3. 违反标记从 `⚠️ 上次违反！`（红警示）改 `〔上一回应这条偏离了，本 turn 看看能否更对齐〕`（合作回顾）
4. preference 文本本身已按维度 A 改造（共情语气）

## 四、关键设计取舍

### 取舍 1：用「默契」替代「规则」会不会让 Agent 不当回事？

**风险**：「默契」语气太软，Agent 觉得「可以违反」。
**缓解**：preference 内容仍强势（「他相信你能 X」「他愿意为 X 等你」），只是包装层共情。工程层 violation_check 不变，硬性拦截仍生效。

### 取舍 2：去掉精确数字会不会让 Agent 不知道边界？

**风险**：没 metric Agent 不知道「中文比例多少够」。
**缓解**：把目标改成「用户读起来不卡」（行为描述），工程层 check 阈值不变（仍 < 40% 拦），但提醒文本不报阈值。

### 取舍 3：加「不需要特意补偿过度」会不会让 Agent 偷懒？

**风险**：Agent 可能解读成「这次违反没事下次也没事」。
**缓解**：保留「上次有偏差，本 turn 调整」明确指出过失，「不需要补偿过度」是阻止防御性堆词（如「真字狂魔」副作用根因）。

### 取舍 4：preference 文本变长会不会增加注入 token 成本？

**当前 sticky preference**平均 70 字，**优化后**平均 100-120 字（共情语气需多点字）。
**影响**：每 turn 注入头部 ~8 条 sticky × 30 字增量 = 240 token 增量。可接受（相对 60K-80K 阈值）。
